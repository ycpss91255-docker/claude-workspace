# ADR-00000001: Why ADR (Architecture Decision Records)

- **Date:** 2026-05-15
- **Status:** Accepted

## Context

Cross-repo work in `ycpss91255-docker` frequently produces
design-discussion conclusions that don't fit any existing artifact
slot:

- **Not a code change** -> not appropriate for commit message or PR
  body. Burying rationale inside a code commit makes it impossible
  to find later via `git log -p`.
- **Not a standing rule** -> not appropriate for `CLAUDE.md`.
  CLAUDE.md is loaded into every session's system prompt; bloat
  has a real cost. Rationale belongs near the decision point, not
  in a perpetually-loaded compendium.
- **Not an actionable issue** -> not appropriate for
  `gh issue create`. Issues track work-to-be-done; concluded
  decisions are work-already-done.

The concrete trigger was a session-long discussion about whether
`base` should grow an entrypoint-switching mechanism (conclusion:
dev style uses `./exec.sh`, service style uses per-repo
customisation of a single `entrypoint.sh`, no multi-entrypoint
folder, no `setup.conf` section). The conclusion was clear,
multi-paragraph, and had nowhere to land.

Six months later someone will ask "why did we decide X instead of
Y" and need to find the rationale. Without a convention, that
rationale exists only in the originating session's transcript --
unfindable, ungrep-able, and lost when the transcript ages out.

## Decision

Adopt the Architecture Decision Record (ADR) convention, lightly
adapted from Michael Nygard's 2011 template.

- **Location:** per-repo `doc/adr/NNNNNNNN-<slug>.md`. Most ADRs
  are repo-specific; no central aggregation site.
- **Numbering:** 8-digit zero-padded, auto-incremented from the
  existing max. Numbers never reused; superseded entries cross-
  link via `Status: Superseded by ADR-NNNNNNNN` on the old one.
- **Template sections:** Date, Status, Context, Decision,
  Alternatives, Consequences. The Alternatives section is explicit
  (beyond Nygard's 4-section minimum) to force capture of the
  trade-offs that drove the choice.
- **Tooling:**
  - `/adr <slug>` slash command + `.claude/scripts/new-adr.sh`
    script generate the file with the template populated.
  - `.claude/hooks/remind_adr_on_design_decision.sh` Stop hook
    nudges the user when a session shows rationale-shaped
    discussion but landed no ADR.

## Alternatives

Three alternatives were considered and rejected:

1. **CLAUDE.md rationale prose.** Rejected: bloats over time. The
   file is already 800+ lines; adding 10 ADRs would push it past
   the point where the agent can hold the whole thing in working
   memory. CLAUDE.md is for *standing rules*, not historical
   decisions.
2. **Commit message bodies.** Rejected: ungrep-able at scale.
   Finding "why did we go with single-entrypoint" requires either
   knowing the exact commit (which you don't, six months later)
   or running `git log --grep` against fuzzy keywords. ADRs live
   in a known location with a known shape.
3. **GitHub Discussions or Wiki.** Rejected: org-level isolation
   between repos works against the per-repo-rationale model. Also,
   `gh` CLI support for Discussions / Wiki is weaker than for
   issues / PRs, and we want the agent to be able to create ADRs
   without leaving the shell.

A fourth option -- a `PostToolUse` hook on `CLAUDE.md` edits that
detects rationale phrasing and nudges to use `/adr` instead --
was considered and dropped. Rationale phrasing legitimately
appears in many CLAUDE.md edits (e.g. discussing why a rule is
written the way it is); the false-positive rate would be too
high. The `Stop` hook reads the *session-level* signal instead.

## Consequences

- Design decisions are findable: a future reader looking for "why
  X" can `ls doc/adr/` in the relevant repo and grep.
- CLAUDE.md stays focused on standing rules + tree-listing +
  workflow contracts. Rationale that catalysed a rule moves to
  the matching ADR; CLAUDE.md links rather than re-derives.
- Adds light overhead per decision: writing 4 sections instead of
  one commit-body paragraph. Tooling (`/adr`) keeps this overhead
  manageable.
- New convention to teach incoming collaborators (human and
  agent). The `/adr` slash command body + this meta ADR are the
  on-ramp.
- Pattern is per-repo; cross-repo aggregation is intentionally
  not in scope. If aggregation later becomes a need, that's its
  own ADR.
